# fix 模式执行流程

> 用途：fix 模式时加载。8 步：评估 → 问题清单 → 分批修复 → 全局验证。

---

**Step 1: 结构扫描 + 自动化 smoke check**
- Glob 目标目录，列出所有文件
- 定位 adapter 共享脚本目录 `core/scripts/`（相对于 adapter 根目录，与 SKILL.md 中 `core/references/` 同级）
- 执行验证脚本（并行，脚本路径使用 `core/scripts/`，不是相对目标 Skill）：
  - `python3 core/scripts/validate-metadata.py --path {目标路径}` → 元数据规范检查
  - `python3 core/scripts/validate-structure.py --path {目标路径}` → 结构 smoke check：模块存在性 + 行数预算 + 目录结构
- 脚本 stderr 中的错误/警告直接纳入问题清单
- 脚本不可用 → 降级为手动检查，标注 "⚠️ 降级: 验证脚本不可用"

失败降级:
- SKILL.md 不存在 → 报告"非 Skill 目录"，建议改用 create 模式
- SKILL.md 无 frontmatter → 标记为 P0 缺陷，继续

✅ Checkpoint: `Step 1 完成: 发现 N 个文件, metadata {PASS/FAIL}, structure {PASS/FAIL}, {X} 行`

**Step 2: 加载检查清单 + 修复约束 + 实战标准**
- Read `core/references/check-guide.md (质量框架部分)`
- Read `core/references/quality-standards.md (反模式部分)`
- Read `core/references/quality-standards.md (完整性部分)`
- Read `core/references/decision-gates.md`
- Read `core/references/practical-best-practices.md`
- Read `core/references/fix-guide.md (修复约束部分)`

失败降级:
- 某文件缺失 → 标注"⚠️ 降级"，继续可用维度

✅ Checkpoint: `Step 2 完成: N/6 个参考文件已加载`

**Step 3: 深度评估**
- 合并执行 check 深度模式的 Steps 3-8
- 产出: 8 模块评分 + 7 反模式风险 + 3 完整性判定 + 发现性验证 + Decision Gate + 实战检查

失败降级:
- 某维度评估失败 → 标注该维度为"降级"，其余正常执行

✅ Checkpoint: `Step 3 完成: 评估完毕, 原始分 X/10`

**Step 4: 输出问题清单**
- 按 P0/P1/P2 排序输出所有发现
- 每项: 优先级 + 问题描述 + 证据位置 + 修复方向
- ⚠️ **审计结果必须先完整输出，再开始修改文件**

完成性要求:
- 问题总数 == Step 3 各维度发现的问题数之和

✅ Checkpoint: `Step 4 完成: P0={a}, P1={b}, P2={c}, 共 {a+b+c} 项`

**Step 5: 修复 P0 问题（逐项修复 + 逐项验证）**

对每个 P0 问题执行以下循环（不可跳过任何一项）：

```
for each P0 issue:
  1. 修改文件（Edit/Write）
  2. 逐项验证本次修改:
     a. 读回修改后的文件，确认修改生效
     b. 变更影响扫描 — 4 种关联查找:
        - 调用方: grep 修改点的名称/路径，检查所有引用处
        - 对称方: 搜索成对出现的另一侧（SKILL.md↔references/、文档↔脚本）
        - 消费方: 搜索引用修改数据的下游（模板变量、报告引用的分数）
        - 同层方: 检查同类模块是否有同样问题
     c. 输出验证结果表:
        | 变更点 | 关联代码 | 是否一致 |
     d. 不一致项 → 当场同步修复（禁止留到后面）
  3. 禁止单边修复 — 改文档必须同步改实现
```

失败降级:
- 无法自动修复 → 标记"需人工处理" + 原因 + 影响范围

✅ Checkpoint: `Step 5 完成: P0 修复 X/{a} 项, 关联同步 N 处, 人工待处理 Y 项`

**Step 6: 修复 P1 问题（同 Step 5 循环）**
- 每项同样执行: 修改 → 读回验证 → 变更影响扫描 → 关联同步

✅ Checkpoint: `Step 6 完成: P1 修复 X/{b} 项, 关联同步 N 处`

**Step 7: 修复 P2 问题（同 Step 5 循环）**
- 每项同样执行: 修改 → 读回验证 → 变更影响扫描 → 关联同步

✅ Checkpoint: `Step 7 完成: P2 修复 X/{c} 项, 关联同步 N 处`

**Step 8: 全局验证 + 修复报告**

8a. 全局一致性验证:
- 规则数/步骤数/checkpoint 总数是否统一
- 引用完整性: 所有提到的文件存在且非空
- 残留扫描: grep 旧路径/旧规则数/旧字段名

8b. 全局变更影响扫描:
- 列出本轮所有变更点的汇总
- 对总变更集执行一次完整的 4 种关联查找
- 确认 Steps 5/6/7 的逐项扫描没有遗漏跨步骤的关联

8c. 交付前 6 条检查清单:
```
□ 验证了吗？— 每项修复都有读回验证 + 影响扫描？
□ 关联检查了吗？— 全局扫描无残留？
□ 是统一方案还是 case-by-case？— 修复方式是否可复用？
□ 写入文件了吗？— 需要持久化的规则是否已落地？
□ 结构合理吗？— 修复后文档结构是否需要重组？
□ 执行边界清晰吗？— 是否有超出审计范围的修改？
```

8d. 输出修复报告（按本文件下方"修复前审计"部分的模板）

8e. **回归评估（必需）**:
- 对修复后的 Skill 重新执行 check 深度模式的 Steps 3-8（8 模块 + 7 反模式 + 3 完整性 + 发现性 + Decision Gate）
- 对比修复前后分数，确认修复确实有效
- 如果修复后分数未提升或下降 → 标记为修复无效，回溯检查修复逻辑

8f. **历史对比 + 写入**: 读取审计历史文件（路径见 `SKILL.md` 审计历史节）查找同一 Skill 的修复前记录（Step 3 的评估分数）。报告中输出: `修复前: X/10 → 修复后: Y/10 (↑+Z)`。报告后先展示待写入记录，仅在用户确认后写入修复后的分数记录。

交叉验证:
- [ ] 已修复数 + 未修复数 == 问题总数（Step 4）
- [ ] 所有修复都有逐项验证结果
- [ ] 所有关联同步都有扫描表证据
- [ ] 回归评估分数 ≥ 修复前分数（修复有效性确认）

✅ Checkpoint: `fix 完成: 修复 X/{total} 项, 关联同步 {M} 处, 人工待处理 Y 项, 回归评估 {修复前}→{修复后}, 全局验证 ✅/❌`

# 修复约束 + 验证方法论

> 用途：fix 模式加载，约束修复行为和验证流程。

---

## 修复前审计（必需）

在任何修改前，先建立完整认知：

### 1. 建立对象清单
- 主入口：SKILL.md
- 参考文档：references/
- 执行脚本：scripts/
- 外围文档：README / 手册等

### 2. 找真值源
统一以下常量到单一来源：
- 规则总数
- 步骤数
- 章节数
- Checkpoint 总数
- 产物命名规则

### 3. 引用检查
- SKILL.md 中提到的每个文件是否**存在**？
- 文件是否**非空**？
- 路径是否**可达**？
- 语义是否**未失效**？

### 4. 承诺-实现检查
- 文档声称的能力，脚本是否真的支持？
- 流程定义的输出结构，脚本是否真的产出？

### 5. 边界检查
- 是否存在互相矛盾的规则？
- 是否同时要求"不要执行"又给出"执行步骤"？

**审计结果必须先输出为问题清单，再开始改文件。**

---

## 5 类工程化失效防护

### 失效 1：真值源失真
- **现象**：规则数/步骤数/checkpoint 总数在多处不一致
- **修复规则**：找到单一真值源，统一所有引用点

### 失效 2：引用断链
- **现象**：SKILL.md/references/README 引用了不存在或空的文件
- **修复规则**：检查每个引用，补齐或删除死链

### 失效 3：承诺高于实现
- **现象**：文档说能做，脚本实际做不到
- **修复规则**：二选一 — 补齐最小可执行实现 / 下调文档承诺

### 失效 4：单边修复
- **现象**：只改文档不改实现，或只改实现不改文档
- **修复规则**：禁止单边修复，改一侧必须同步改另一侧

### 失效 5：无验证收口
- **现象**：修完未做语法检查、smoke test、回归核对
- **修复规则**：修复后必须验证，不可只声明"已修复"

---

## 修复约束（硬性）

1. **审计先于修改** — 先输出问题清单，再开始改文件
2. **禁止单边修复** — 改文档必须同步改实现
3. **按优先级分批修** — P0 → P1 → P2，每批修完立即局部验证
4. **空文件/死链接视为缺陷** — 不可忽略
5. **修复范围仅限审计发现** — 不可"顺便优化"审计未提及的内容

---

## 优先级定义

| 级别 | 标准 | 示例 |
|------|------|------|
| **P0** | 导致错误路由/错误结论/断链/错误执行 | 真值源失真、核心流程缺失、空关键文件 |
| **P1** | 导致行为不稳/职责冲突/验证失真 | 环境检测失真、参数 bug、兼容性问题 |
| **P2** | 导致维护成本上升/说明不准/文档漂移 | 输出质量、结构化程度、可维护性 |

---

## 验证方法论

修复完成后按以下顺序验证：

### 1. 一致性验证
- 规则总数、章节数、checkpoint 总数是否统一
- 引用文件是否存在且非空

### 2. 语法与兼容性验证（如有脚本）
- `bash -n` / `zsh -n` 语法检查
- shebang 与特性是否匹配
- 执行权限是否正确

### 3. 最小功能 smoke test
- 至少一个最小输入
- 至少打通一条关键路径

### 4. 结果可信度验证
- 失败不误判为成功
- 空结果不伪造成发现
- 计数不失真

### 5. 残留扫描
- grep/rg 搜索旧路径、旧规则数、旧字段名是否仍残留
