# check 模式执行流程

> 用途：check 模式时加载。支持快速检查（3 步）和深度检查（9 步）。

---

## 快速检查（3 步）

**Step 1: 结构扫描 + 自动化 smoke check**
- Glob 目标目录，列出所有文件
- 先定位当前 `skill-craft` 的根目录（本 `SKILL.md` 所在目录，记为 `{checker-root}`）
- 执行当前 Skill 自带验证脚本（并行，脚本路径相对 `{checker-root}`，不是相对目标 Skill）：
  - `python3 {checker-root}/scripts/validate-metadata.py --path {目标路径}` → 检查 name/description 规范
  - `python3 {checker-root}/scripts/validate-structure.py --path {目标路径}` → 结构 smoke check：模块存在性 + 行数预算 + 目录结构
- 脚本输出直接纳入报告：exit 0 = 通过，exit 1 = stderr 中的错误/警告作为发现项
- 如脚本不存在或执行失败 → 降级为手动检查（Glob + Read frontmatter），标注 "⚠️ 降级: 验证脚本不可用"

失败降级:
- SKILL.md 不存在 → 报告"非 Skill 目录"，终止检查
- SKILL.md 无 frontmatter → 标记为结构缺陷，继续检查正文

✅ Checkpoint: `Step 1 完成: 发现 N 个文件, metadata {PASS/FAIL}, structure {PASS/FAIL}, {X} 行`

**Step 2: 模块存在性扫描**
- Read SKILL.md
- 对 8 个模块逐个检查关键结构元素是否存在（标题/关键词/表格）
- 仅检查存在性，不评估质量

失败降级:
- SKILL.md 过大无法一次读取 → 分段读取
- 模块标题使用非标准命名 → 按内容语义匹配（而非仅匹配标题关键词）

✅ Checkpoint: `Step 2 完成: N/8 模块存在`

**Step 3: 快速报告**
- 生成缩略报告：结构 + 模块存在矩阵 + Top 3 缺口
- 给出粗略评分和建议（是否需要深度检查）

✅ Checkpoint: `快速检查完成: 粗略评分 N/10`

---

## 深度检查（9 步）

**Step 1: 结构扫描**（同快速 Step 1）

**Step 2: 加载检查清单**
- Read `core/references/check-guide.md (质量框架部分)`
- Read `core/references/quality-standards.md (反模式部分)`
- Read `core/references/quality-standards.md (完整性部分)`
- Read `core/references/decision-gates.md`
- Read `core/references/practical-best-practices.md`

失败降级:
- 某个检查清单文件缺失 → 标注"⚠️ 降级: 缺少 {文件名}，该维度无法评估"，继续其他维度

✅ Checkpoint: `Step 2 完成: N/5 个检查清单已加载`

**Step 3: 8 模块深度评估**
- Read 目标 Skill 的全部文件（SKILL.md + references/ + scripts/）
- 按本文件下方"Quality Framework"部分的检查项逐模块评估
- 每模块: 分数(0-2) + 证据引用 + 具体问题

完成性要求:
- 输入: 8 个模块检查清单
- 必须处理: 全部 8 个模块（模块 7 可标 N/A）
- 验收条件: 已评估模块数 == 8

失败降级:
- 目标文件无法读取 → 标注"⚠️ 降级: {文件} 不可读"，基于已读内容评估
- 模块内容无法明确判断 → 标注"UNABLE TO ASSESS: [原因]"

✅ Checkpoint: `Step 3 完成: 8 模块已评估, 原始分 X/16`

**Step 4: 7 反模式脆弱性评估**
- 按 `quality-standards.md` 反模式部分的防御/脆弱性指标逐项检查
- 每项: 风险级别(Low/Med/High) + 证据引用

完成性要求:
- 输入: 7 个反模式检查项
- 必须处理: 全部 7 项（模式 5 可标 N/A）
- 验收条件: 已评估数 == 7

失败降级:
- 无法判断某模式风险 → 标注"UNABLE TO ASSESS"，不标为 Low

✅ Checkpoint: `Step 4 完成: 7 反模式已评估, N 项 High 风险`

**Step 5: 3 完整性原则检查**
- 按 `quality-standards.md` 完整性部分逐项判定 Pass/Partial/Fail
- 每项: 状态 + 证据

失败降级:
- 流程步骤无法识别 → 标注 Fail + "无法识别流程步骤结构"

✅ Checkpoint: `Step 5 完成: 3 原则已评估`

**Step 6: 实战最佳实践检查**
- 按 `practical-best-practices.md` 检查:
  - description 质量（触发短语 ≥5 + 负面边界）
  - 信号路由优先级（现象信号 > 关键词）
  - Token 预算（SKILL.md ≤200 行？reference 按场景拆分？）
  - 10 条反模式对照（重复段、Action 嵌套、写入无前置检查等）
- 每项: 通过/未通过 + 证据

✅ Checkpoint: `Step 6 完成: 实战检查 X/N 项通过`

**Step 7: 发现性验证（Discovery Validation）**
- 基于目标 Skill 的 frontmatter description + 触发条件，生成 2 组测试 prompt：
  - **应触发组**（3 条）：模拟真实用户会发出的、应当激活该 Skill 的 prompt
  - **不应触发组**（3 条）：模拟容易误触发的、相似但不属于该 Skill 范围的 prompt
- 逐条分析 description 对每条 prompt 的匹配度：
  - 应触发 prompt 是否能被 description 中的关键词/短语覆盖？
  - 不应触发 prompt 是否能被 "Do not use" / "不触发" 条件排除？
- 输出发现性评估矩阵：

```
| # | Prompt | 类型 | 匹配结果 | 问题 |
|---|--------|------|---------|------|
| 1 | "检查这个 skill 质量" | 应触发 | ✅ 命中 | — |
| 2 | "review this code" | 不应触发 | ❌ 未排除 | description 缺少 code review 排除 |
```

- 发现性评分规则：
  - 6/6 全部正确 → Pass
  - 4-5/6 正确 → Partial（生成 P1 行动项：补充缺失的触发/排除短语）
  - ≤3/6 正确 → Fail（生成 P0 行动项：重写 description）

失败降级:
- 目标 Skill 无 frontmatter description → 直接标记 Fail + P0 行动项"缺少 description"
- description 过短（<50 字符）→ 标记 Partial + P1 行动项"description 信息不足"

✅ Checkpoint: `Step 7 完成: 发现性验证 X/6 正确, 评级 {Pass/Partial/Fail}`

**Step 8: Decision Gate 评估**
- 按 `decision-gates.md` 检查目标 Skill 是否把 signal / evidence / counter-evidence / decision 分开
- 抽样 3-5 类目标 Skill 可能输出的强结论，识别 claim_type、required evidence、counter-evidence 与 completeness ceiling
- 每类: Pass/Partial/Fail + 证据引用 + 是否存在 signal-only decision 或 unchecked counter-evidence 风险

完成性要求:
- 输入: 目标 Skill 的主要强结论类型
- 必须处理: 至少 3 类；少于 3 类时全部处理并说明原因
- 验收条件: 已评估 claim type 数 == 计划评估数

失败降级:
- 目标 Skill 不输出判断/建议/评分/改动/风险结论 → 标记 N/A
- 无法判断 claim type → 标注 UNABLE TO ASSESS，不得推测为 Pass

**DG 10% 权重分数计算公式（硬性，避免主观打分）**:
- 每类评估结果: Pass = 10 分, Partial = 5 分, Fail = 0 分, UNABLE TO ASSESS = 0 分（不得推测）
- DG 归一化分 = (Σ 各类得分) / (已评估类数 × 10)  → 落在 [0.0, 1.0]
- DG 加权贡献 = DG 归一化分 × 1.0（= 总分 10 分制的 10%）
- **N/A 场景**（Skill 不输出任何强结论）: DG 权重回冲给模块分（模块权重 55% → 65%）；报告注明 "DG=N/A, 权重回冲至模块"
- **counter-evidence 未检查**: 该类自动降为 Partial（不得评 Pass）

✅ Checkpoint: `Step 8 完成: Decision Gate 评估 X/Y 类, signal-only 风险 A 项, counter-evidence 缺失 B 项, DG 加权 Z/10%`

**Step 9: 综合报告**
- 按 `core/references/report-template.md` 格式生成完整报告
- 计算加权总分: 模块(55%) + 反模式(20%) + 完整性(15%) + Decision Gate(10%)
- 实战检查结果作为附加评分
- 生成 P0/P1/P2 行动项

**封顶规则（Hard Cap）** — 命中任一项，加权总分 clamp 为 `≤ 6.0/10`，并在报告顶部标注 "⚠️ 封顶触发: [第 N 条] - [原因]":

| # | 触发条件 | 证据来源 |
|---|---------|---------|
| 1 | 触发条件互斥冲突（同一 input 同时命中多条路径且无优先级定义） | Step 3 模块 1 发现 |
| 2 | Decision Gate Fail 项 ≥ 2 | Step 8 评估结果 |
| 3 | 核心 references/ 或 scripts/ 断链（frontmatter tools 或 SKILL.md 正文引用但路径不存在/为空） | Step 1 smoke check + Step 2 加载结果 |
| 4 | validate-metadata.py 或 validate-structure.py FAIL | Step 1 脚本 exit code |
| 5 | quick check 的结果冒充 deep check（实际步数 <9 但输出 deep quality score） | 步数计数 |

封顶生效后:
- 加权总分 = `min(原始加权总分, 6.0)`
- 所有 `accepted` 结论降为 `tentative`（与 `decision-gates.md` Report Gate #6 对齐）
- 报告顶部强制输出 "⚠️ 封顶触发: [第 N 条] - [原因一句话]"
- 不影响 P0/P1/P2 行动项生成；封顶是评分层规则，不是执行终止条件

**产物契约附加观察（不计分）** — 在报告中显式回答以下 3 项，标注 `OBSERVED / MISSING / N/A` + 证据引用（file:line）:

- [ ] 目标 Skill 是否定义稳定输出（文件名、目录结构、schema/格式）？
- [ ] 是否有唯一真值源（同一产物不在多处被原始定义，派生产物除外）？
- [ ] 是否区分 human-readable / machine-readable / work artifacts？

> 本节当前为**样本收集阶段**（需 2-3 次真实审计数据）。**不计入加权总分**；仅在报告中显式呈现产物侧视角。升格为正式模块的决策推迟到有足够样本后。

- **历史对比**: 读取审计历史文件（路径见 `SKILL.md` 审计历史节），查找同一 Skill（按 path 匹配）的历史记录。有历史时在报告顶部输出趋势行: `评分趋势: 5.2 (2026-03-10) → 7.2 (本次) ↑+2.0`。无历史则不提及。

交叉验证:
- [ ] 行动项总数 == Steps 3+4+5+6+7+8 发现的问题数之和
- [ ] 所有行动项都有 file:location 引用

交付前检查（自动输出）:
```
□ 验证了吗？□ 关联检查了吗？□ 统一方案？□ 结构合理？□ 执行边界清晰？
```

**写入历史**: 报告输出后，先展示待写入记录；仅在用户确认后追加到审计历史文件。

✅ Checkpoint: `深度检查完成: 总分 N/10 (封顶 {是/否, 触发 #X / —}), 发现性 {Pass/Partial/Fail}, Decision Gate {Pass/Partial/Fail/N/A}, 实战检查 X/N, 产物契约 {OBSERVED/MISSING/N/A}×3, 可移植性 {PASS/WARN/FAIL}×4, P0: X, P1: Y, P2: Z 项`

# Quality Framework: 8 模块检查清单

> 用途：check/fix 模式加载，逐项评估目标 Skill 的模块完整性。

## 评分标准

- **0 = ABSENT**：模块完全不存在
- **1 = PARTIAL**：部分要素存在，关键项缺失
- **2 = COMPLETE**：所有必需要素存在且结构良好

### 评分校准示例（降低判断偏差）

**模块 1 触发条件 — 0 分案例**：
```yaml
description: 一个代码审计工具
```
无触发条件、无不触发、无歧义处理。

**模块 1 触发条件 — 1 分案例**：
```yaml
description: 代码安全审计。当用户要求审计代码时使用。
```
有触发条件，但缺"不触发"反例和歧义处理。

**模块 1 触发条件 — 2 分案例**：
```yaml
description: |
  代码安全审计。用户提供代码目录 + 要求审计/扫描/检查安全性时触发。
  DO NOT trigger: 讨论安全概念、非代码文件审计、runtime debugging。
```
多条件 AND + 不触发反例 + 输入类型限定。

**模块 4 输出约束 — 0 分案例**：无任何输出限制。

**模块 4 输出约束 — 1 分案例**：`"保持简洁"`（笼统，无具体禁止项）。

**模块 4 输出约束 — 2 分案例**：
```markdown
禁止输出:
- 开场白（"让我来分析..."）
- 工具调用描述（"我将使用 Read..."）
- 已知信息复述
```
≥3 条具体禁止项。

**模块 2 行为准则 — 0 分案例**：无任何行为规则或约束。

**模块 2 行为准则 — 1 分案例**：
```markdown
## 规则
- 输出必须包含文件路径
- 不要修改未提及的文件
```
有规则，但无 ❗ 标记、无"每次输出前自检"、无"会话全程有效"声明。

**模块 2 行为准则 — 2 分案例**：
```markdown
## 行为准则（整个会话有效，不因对话长度放松）
1. ❗ 每个发现必须引用来源 — 每次输出前自检
2. ❗ 评分基于检查清单 — 每次输出前自检
```
有 ❗ 标记（≤3 条）+ 自检机制 + 全程有效声明。

**模块 3 工具优先级 — 0 分案例**：无工具相关说明。

**模块 3 工具优先级 — 1 分案例**：
```markdown
优先使用 Read 工具读取文件，失败时用 cat。
```
有工具说明，但非表格格式，降级条件模糊（"失败时"而非"连续 N 次"）。

**模块 3 工具优先级 — 2 分案例**：
```markdown
| 操作 | 首选 | 降级条件 | 降级工具 |
|------|------|---------|---------|
| 读文件 | Read | Read 返回错误 | Bash cat |
| 搜索 | Grep | 连续 2 次失败 | Bash grep |
- 单次超时 ≠ 不可用，必须重试
- 降级必须标注 "⚠️ 降级: [原因]"
```
表格格式 + 具体降级条件 + 禁止自行降级。

**模块 5 流程 Checkpoint — 0 分案例**：`"分析代码，输出报告。"` 无步骤、无 Checkpoint。

**模块 5 流程 Checkpoint — 1 分案例**：
```markdown
Step 1: 扫描文件
Step 2: 分析问题
Step 3: 输出报告
```
有步骤编号，但无 Checkpoint、无等式验收、无失败路径。

**模块 5 流程 Checkpoint — 2 分案例**：
```markdown
Step 1: 扫描文件
  完成性要求: 已扫描数 == 目录下文件总数
  失败降级: Glob 失败 → Bash ls
  ✅ Checkpoint: "Step 1 完成: 扫描 X/Y 文件"
```
步骤 + 等式 + Checkpoint + 失败路径全具备。

**判分规则**：先数必需要素的有/无（客观），再判结构质量（主观但有锚点）。有争议时偏向低分——宁可漏给分，不可错给分。

## 模块 1：触发条件（防模式7：触发模糊）

必需要素：
- [ ] **触发条件**：使用多条件 AND 组合（非单一关键词匹配）
- [ ] **不触发**：列出 ≥2 个容易误触发的反例场景
- [ ] **歧义处理**：定义了无法判断时的具体澄清问题

加分要素（有 → 质量更高）：
- [ ] 区分全链路触发 vs 单点触发（如："全链路分析" vs "只查壳"）
- [ ] 意图分类（执行/分析/生成 vs 讨论/提问/引用）

判定：
- 3 项必需全有 → 2 分
- 有触发条件但缺不触发或歧义处理 → 1 分
- 无触发条件定义 → 0 分

---

## 模块 2：行为准则 + 抗衰减锚点（防模式2：约束衰减）

必需要素：
- [ ] **❗ 标记规则**：最关键的 2-3 条规则标记为 ❗ + "每次输出前自检"
- [ ] **规则数量控制**：❗ 标记的规则 ≤ 3 条（太多等于没标）
- [ ] **会话全程有效声明**：明确规则"不因对话长度而放松"

判定：
- 有 ❗ 规则且 ≤3 条 + 全程有效声明 → 2 分
- 有规则但无 ❗ 标记或无全程有效声明 → 1 分
- 无行为准则 → 0 分

---

## 模块 3：工具优先级表（防模式3：工具选择漂移）

必需要素：
- [ ] **表格格式**：操作 | 首选工具 | 降级条件 | 降级工具
- [ ] **降级条件具体**：使用"连续 N 次失败"而非模糊的"失败时"
- [ ] **禁止未经确认的自行降级**
- [ ] **单次超时 ≠ 工具不可用**：要求重试

判定：
- 有表格 + 具体降级条件 + 禁止自行降级 → 2 分
- 有工具说明但非表格或降级条件模糊 → 1 分
- 无工具优先级定义 → 0 分

---

## 模块 4：输出约束（防模式4：输出膨胀）

必需要素：
- [ ] **禁止输出列表**：≥ 3 条具体的"禁止 X"（非笼统"保持简洁"）
- [ ] **开场白禁止**：禁止"让我来分析..." / "首先我们需要..."
- [ ] **工具调用描述禁止**：禁止"我将使用 X 工具"
- [ ] **领域相关输出限制**：如代码注释、报告章节等的具体约束

判定：
- ≥3 条具体禁止项 + 领域约束 → 2 分
- 有输出约束但禁止项 <3 或过于笼统 → 1 分
- 无输出约束 → 0 分

---

## 模块 5：流程步骤 + Checkpoint（防模式1：完整性跳过）

必需要素：
- [ ] **编号步骤**：有明确的 Step 1/2/3... 或阶段划分
- [ ] **等式验收**：可数产物的步骤有 `已完成数 == 应完成数` 等式
- [ ] **Checkpoint 输出**：每步有 ✅ Checkpoint + 量化指标
- [ ] **失败降级路径**：每步定义了失败时的处理方式（非只有正常路径）

判定：
- 有步骤 + 等式 + Checkpoint + 失败路径 → 2 分
- 有步骤但缺等式/Checkpoint/失败路径中的 ≥2 项 → 1 分
- 无流程步骤定义 → 0 分

---

## 模块 6：依赖链声明（防模式5：依赖链断裂）

必需要素：
- [ ] **数据传递声明**：明确标注"Step N 的输入 = Step M 的完整产出"
- [ ] **禁止重新生成**：要求后续步骤引用前序实际输出（不可重新搜索）
- [ ] **Checkpoint 计数链**：后步引用前步的计数值
- [ ] **交叉验证**：写入文件前数字一致性检查

判定：
- 有数据传递声明 + 禁止重新生成 + 交叉验证 → 2 分
- 有部分依赖说明但不完整 → 1 分
- 无依赖链声明 → 0 分

---

## 模块 7：子 Agent 委派规则（防模式6：并行孤岛）

必需要素：
- [ ] **委派 prompt 复制原始约束**（不转述）
- [ ] **完成性约束**："输入集共 N 项，必须全部处理"
- [ ] **分工边界**："你只负责 X，不要涉及 Y"
- [ ] **合并检查清单**：去重 + 一致性 + Checkpoint 计数之和 == 总任务量

**N/A 条件**：Skill 步骤数 ≤2 且无并行执行需求时标记 N/A，不扣分。

判定：
- 有委派规则 + 合并检查 → 2 分
- 有基本委派但缺合并检查 → 1 分
- 需要但缺失 → 0 分
- 不需要 → N/A

## 模块 8：零结果与幻觉防护（防模式8：幻觉填充）

必需要素：
- [ ] **来源引用要求**：所有结论必须引用具体来源（工具名+数据 / 文件路径+行号）
- [ ] **无来源禁止输出**："无引用来源的结论 = 不允许输出"
- [ ] **零结果处理表**：场景 | 正确输出 | 禁止输出
- [ ] **标注分级**：区分确认结果（无标注）/ 降级分析（⚠️）/ 通用建议（💡）

判定：
- 有来源要求 + 零结果表 + 标注分级 → 2 分
- 有部分幻觉防护但不完整 → 1 分
- 无幻觉防护 → 0 分

## 结构检查（额外 5 项）

来源：实战教训总结，与 8 模块并列检查。

- [ ] **无重复段**：同一规则不在两个段中出现（如"行为准则"和"完整规则"不重叠）
- [ ] **Action 独立**：执行逻辑独立成段，不嵌在输出格式模板里
- [ ] **边界覆盖**：无参数/无输入时有明确处理规则
- [ ] **写入前检查**：涉及写文件的操作有 先读→去重→验证 流程
- [ ] **多 Skill 统一**：同项目的多个 Skill 结构一致（如有）

每项有=+0.2 分附加，最高 +1.0 分（不改变满分 10 分上限，用于区分同分 Skill）
