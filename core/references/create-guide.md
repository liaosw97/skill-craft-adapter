# create 模式执行流程

> 用途：create 模式时加载。6 步：需求确认 → 规模判断 → 生成 → 自检 → 自动化验证 → 报告。

---

**Step 1: 明确需求**
- 确认 Skill 用途、领域、输入物类型、输出物类型
- 确认目标目录路径
- 如信息不足，询问用户补充

失败降级:
- 用户描述过于模糊 → 提出 3 个具体问题引导：(1) 输入是什么？(2) 输出是什么？(3) 核心步骤有几个？
- 目标目录已存在 SKILL.md → 建议改用 fix 模式

✅ Checkpoint: `Step 1 完成: 领域={X}, 输入={Y}, 输出={Z}, 路径={P}`

**Step 2: 加载生成指南 + 规模判断**
- Read `core/references/create-guide.md (生成指南部分)`
- 根据预估步骤数判断复杂度: 轻量(1-2步) / 中等(3-5步) / 重型(6+步)
- 确定必需模块列表

失败降级:
- 本文件下方"生成指南"部分 不可读 → 使用内置规模适配表（轻量: 模块1+2+4+8, 中等: 1-5+8, 重型: 全部 8 个）

✅ Checkpoint: `Step 2 完成: 复杂度={X}, 必需模块={列表}`

**Step 3: 生成 Skill 文件**
- 基于 `core/references/skill-scaffold.md` 骨架 + `本文件下方"生成指南"部分` 模板
- 创建目录结构: SKILL.md + references/ + scripts/（按需）
- 按规模适配填充必需模块，省略的模块不保留占位
- 如果 Skill 会输出判断、建议、评分、改动或风险结论，必须按 `core/references/decision-gates.md` 添加领域化 Decision Gate；不输出强结论时显式标 N/A

完成性要求:
- 必需模块全部填充（非占位）
- 已填充模块数 == 必需模块数（Step 2 确定的列表）
- 每个填充的模块包含领域相关的具体内容（非通用模板文字）
- Decision Gate 不得只列 signal candidate，必须列 required evidence、counter-evidence、completeness ceiling 与降级规则

失败降级:
- 领域知识不足以填充某模块 → 标注"⚠️ 需用户补充: [具体需要什么信息]"

✅ Checkpoint: `Step 3 完成: 生成 N 个文件, 填充 M/{必需模块数} 模块`

**Step 4: 自检清单验证**
- 按 `本文件下方"生成指南"部分` 末尾的自检清单逐项检查
- 模块检查（8 项）+ 结构检查（5 项）

完成性要求:
- 已检查项数 == 清单总数
- 验收条件: 通过数 == 总数（未通过项必须修正后重检）

失败降级:
- 自检未通过 → 修正 → 重检 → 仍未通过 → 标记原因 + 建议人工处理

✅ Checkpoint: `Step 4 完成: 自检 X/Y 项通过`

**Step 5: 自动化验证**
- 运行 `python3 {checker-root}/scripts/validate-metadata.py --path {skill-dir}` 验证 name + description
- 运行 `python3 {checker-root}/scripts/validate-structure.py --path {skill-dir}` 验证目录结构 + 8 模块存在性
- 两个脚本均 PASS → 继续；有 ERROR → 修正后重跑

失败降级:
- 脚本不可用 → 手动检查 name(kebab-case) + description(≥50字符) + 目录结构
- 标注 "⚠️ 降级: 自动化验证不可用，已手动检查"

✅ Checkpoint: `Step 5 完成: metadata {PASS/FAIL}, structure {PASS/FAIL}`

**Step 6: 输出创建报告**
- Skill 概述（名称、用途、复杂度、模块数）
- 文件清单
- 自检结果（Step 4）+ 自动化验证结果（Step 5）
- 使用说明（如何触发这个新 Skill）

✅ Checkpoint: `create 完成: {skill-name} 已生成, 自检 X/Y 通过, 自动验证 PASS/FAIL`

# 8 模块生成指南

> 用途：create 模式加载，指导从零生成新 Skill。

---

## 规模适配表

根据 Skill 复杂度选择必需模块：

| 复杂度 | 步骤数 | 必需模块 | 可省略模块 |
|--------|--------|---------|-----------|
| **轻量**（单步骤工具调用） | 1-2 | 1 + 2 + 4 + 8 | 3, 5, 6, 7 |
| **中等**（线性流程） | 3-5 | 1 + 2 + 3 + 4 + 5 + 8 | 6, 7（按需） |
| **重型**（多阶段+子agent+验证） | 6+ | 全部 8 个模块 | 无 |

**强制必需判断**：
- 产出数量可数 → 模块 5（Checkpoint + 等式）不可省
- 多步骤且后步依赖前步 → 模块 6（依赖链）不可省
- 使用子 agent → 模块 7（委派规则）不可省
- 调用外部工具/MCP → 模块 3（工具优先级）+ 模块 8（幻觉防护）不可省

---

## 模块 1：触发条件模板

```markdown
## 触发条件

触发（同时满足）:
- 用户提供了 [具体输入物类型]
- 用户意图是 [执行/分析/生成]（非讨论/提问/引用）

不触发:
- [场景1：容易误触发的]
- [场景2：相似但不同的]

歧义处理:
- 无法判断时询问: "[具体的澄清问题]"
```

---

## 模块 2：行为准则模板

```markdown
## 行为准则

以下规则在整个会话期间有效，不因对话长度而放松：

1. ❗ [最关键规则] — 每次输出前自检
2. ❗ [第二关键规则] — 每次输出前自检
3. [其他规则]
```

**要点**：❗ 标记 ≤ 3 条，选择最容易被遗忘的规则。

---

## 模块 3：工具优先级模板

```markdown
## 工具优先级（不可自行降级）

| 操作 | 首选工具 | 降级条件 | 降级工具 |
|------|---------|---------|---------|
| [操作A] | [专用工具] | 连续 2 次失败 | [通用替代] |
| [操作B] | [专用工具] | Step X 确认不可用 | [通用替代] |

- 单次超时 ≠ 工具不可用，必须重试 1 次
- 降级必须标注: "⚠️ 降级: [原因]"
```

---

## 模块 4：输出约束模板

```markdown
## 输出约束

禁止输出:
- "让我来分析一下..." / "首先我们需要..." 等开场白
- 工具调用前后的重复描述（直接调用，不说"我将使用 X 工具"）
- 已知信息的复述（用户刚说的话、文件刚读的内容）
- [领域相关的冗余模式]

代码/产物约束:
- [具体限制]
```

**要点**：禁止项 ≥ 3 条，用"禁止 X"而非"保持简洁"。

---

## 模块 5：流程步骤模板

```markdown
## 执行流程

### Step 1: [动作]
[操作指令]

完成性要求:
- 输入集: [来源和数量]
- 必须处理: 每个/全部/≥N%
- 验收条件: 已处理数 == 输入总数

失败降级:
- [操作A] 失败 → [操作B] → B 也失败 → 记录原因，不允许跳过

✅ Checkpoint: `Step 1 完成: [量化指标]`
```

---

## 模块 6：依赖链模板

```markdown
## 依赖链

Step 2 的输入 = Step 1 的完整产出（不可重新搜索）
Step 3 的输入 = Step 1 + Step 2 的累计产出

Checkpoint 传递:
- Step 1: "完成: N 项"
- Step 2: "输入 N 项（来自 Step 1），新增 M 项，累计 N+M 项"

交叉验证（写入文件前）:
- [ ] 产物中的数字 == 各步骤 Checkpoint 累计值
```

---

## 模块 7：子 Agent 委派模板

```markdown
## 子 Agent 委派

委派 prompt 必须包含:
1. 完整的任务清单和 Checkpoint 要求（复制原文，不转述）
2. 完成性约束: "输入集共 N 项，必须全部处理"
3. 结果格式: "结果末尾输出 Checkpoint: 已处理 X/N 项"

并行规则:
- 有数据依赖 → 不可并行
- 明确边界: "你只负责 X，不要涉及 Y"

合并检查:
- [ ] 去重
- [ ] 一致性
- [ ] Checkpoint 计数之和 == 总任务量
```

---

## 模块 8：幻觉防护模板

```markdown
## 事实性约束

所有结论必须引用来源（工具名 + 数据 / 文件路径 + 行号）。
无来源引用的结论 = 不允许输出。

| 场景 | 正确输出 | 禁止输出 |
|------|---------|---------|
| 搜索返回 0 条 | "PASS: 未发现" | 编造发现 |
| 工具不可用 | "⚠️ 未检测（工具不可用）" | 编造分析 |
| 数据不足 | "需进一步验证" | 确定性结论 |

标注:
- 工具确认 → 无标注
- 降级推测 → "⚠️ 降级分析"
- 领域建议 → "💡 通用建议（非实际检测结果）"
```

---

## 生成后自检清单

生成完后逐项检查（不可跳过）：

```
模块检查:
[ ] 模块 1: 触发 + 不触发 + 歧义处理？
[ ] 模块 2: ≤3 条 ❗ 抗衰减规则？
[ ] 模块 3: 操作/首选/降级条件/降级工具表格？
[ ] 模块 4: ≥3 条"禁止输出"项？
[ ] 模块 5: 可数步骤有等式 + Checkpoint + 失败路径？
[ ] 模块 6: 步骤间数据传递显式声明 + 交叉验证？
[ ] 模块 7: 委派复制原始约束 + 合并检查？
[ ] 模块 8: 来源引用 + 零结果处理？

结构检查:
[ ] 无重复段
[ ] Action 独立
[ ] 边界覆盖
[ ] 写入前检查
[ ] 多 Skill 统一
```
