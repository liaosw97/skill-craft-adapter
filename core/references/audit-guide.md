# audit 模式执行流程

> 用途：audit 模式时加载。6 步：系统扫描 → 路由审计 → 一致性审计 → 验证闭环 → 报告。

---

**Step 1: 系统扫描**
- Glob 父目录，找到所有 `*/SKILL.md`
- 列出所有 Skill 及其文件清单
- 识别系统级文档（README、手册、测试清单）

失败降级:
- 未找到任何 SKILL.md → 报告"非 Skill 系统目录"，终止
- 仅找到 1 个 SKILL.md → 建议改用 check 模式

✅ Checkpoint: `Step 1 完成: 发现 N 个 Skill, M 个系统级文档`

**Step 2: 加载审计标准**
- Read `references/audit-guide.md (审计标准部分)`

失败降级:
- 文件不可读 → 使用内置 9 项必查清单（路由边界、角色分工、真值源、引用链、承诺实现、旧口径、验证闭环、Decision Gate、外围文档）

✅ Checkpoint: `Step 2 完成: 审计标准已加载, 9 项必查`

**Step 3: 路由与分工审计**
- 对比所有 Skill 的 description + 触发条件 + DO NOT trigger
- 检查路由边界是否互斥
- 检查角色分工是否清晰、有无越界

完成性要求:
- 输入: N 个 Skill 的触发条件
- 必须: 两两对比所有组合
- 验收: 已对比组合数 == N*(N-1)/2

失败降级:
- 某 Skill 的 SKILL.md 无法读取 → 标注该 Skill 为"不可审计"，其余正常
- 触发条件非标准结构 → 基于 description 字段做最佳努力对比

✅ Checkpoint: `Step 3 完成: 检查 X 个路由组合, 发现 Y 个冲突`

**Step 4: 一致性、完整性与 Decision Gate 审计**
- 真值源统一: grep 数字常量，验证多处引用一致
- 引用链完整: 提取所有文件路径引用，验证存在且非空
- 承诺-实现: 文档声称 vs 脚本能力
- 旧口径残留: rg 搜索已知旧值
- Decision Gate: 对会输出强结论的 Skill，检查是否定义 signal/evidence/counter-evidence/decision、completeness ceiling 与降级规则

完成性要求:
- 必查项: 真值源 + 引用链 + 承诺实现 + 旧口径 + Decision Gate (5 项)
- 验收: 已检查项数 == 5

失败降级:
- 无脚本目录 → 承诺-实现检查标注 N/A
- 无法推断旧值 → 旧口径检查标注"无基线，跳过"

✅ Checkpoint: `Step 4 完成: 5 项一致性检查, 发现 X 个问题`

**Step 5: 验证闭环与外围文档审计**
- 测试清单是否存在、是否与当前实现一致
- 外围文档是否仍传播旧规则
- 设计初衷判断（如有明确设计目标）

失败降级:
- 无测试清单 → 标注"验证闭环: 不存在"（本身即为发现）
- 无外围文档 → 标注 N/A

✅ Checkpoint: `Step 5 完成: 验证闭环 + 外围文档检查`

**Step 6: 系统审计报告**
- 按 P0/P1/P2 排序输出所有发现
- 每项含: 涉及的 Skill + 证据位置 + 修复建议
- 是否满足原始设计初衷（如适用）
- **评分趋势附表**: 读取审计历史文件（路径见 `SKILL.md` 审计历史节），对有历史记录的 Skill 附"评分趋势"表（Skill名 | 上次分数 | 本次分数 | 变化）。无历史的 Skill 不列入。

交叉验证:
- [ ] 发现总数 == Steps 3+4+5 的问题数之和

**写入历史**: 为每个被审计的 Skill 先展示待写入记录；仅在用户确认后追加到审计历史文件（mode 为 "audit"）。

✅ Checkpoint: `audit 完成: P0={a}, P1={b}, P2={c}, 涉及 {N} 个 Skill`

# 多 Skill 系统审计标准

> 用途：当目标不是单个 Skill，而是包含多个 Skill 的系统（多个 SKILL.md + README + 手册等）时加载。
>
> 触发条件：用户指定了包含多个 Skill 的父目录，或明确要求"系统级审计"。

---

## 审计范围

至少覆盖以下对象：
- 所有 `*/SKILL.md`
- 每个 Skill 的 `references/`
- 每个 Skill 的 `scripts/`
- 系统级 `README.md`
- 手册 / 速查文档 / 全链路方案文档
- 实战经验目录
- 测试清单 / 用例文档

---

## 必查 9 项

### 1. 路由边界
- 多个 Skill 的 description / DO NOT trigger 是否**互斥**？
- 是否存在职责重叠或抢路由？
- 检查方法：对比所有 Skill 的 frontmatter description + 触发条件章节，找交集

### 2. 角色分工
- 各 Skill 的定位是否清楚（如：查 / 验 / 改）？
- 是否有某个 Skill 越界承担别的 Skill 的职责？
- 检查方法：列出各 Skill 的核心动作词（评估/修复/创建/扫描...），验证无重叠

### 3. 真值源统一
- 规则数 / 步骤数 / 章节数 / checkpoint / 输出字段在多个文件中是否一致？
- 检查方法：grep 数字常量，验证所有引用点的值一致

### 4. 引用链完整
- SKILL → reference → scripts → README / 手册 是否有断链？
- 是否引用了不存在文件、旧文件名、空文件？
- 检查方法：提取所有文件路径引用，逐个验证存在性和非空性

### 5. 承诺-实现一致
- README / 手册 / SKILL 声称的能力，脚本是否真的支持？
- reference 中定义的输出结构，脚本是否真的产出？
- 检查方法：对比文档声称 vs 脚本实际代码路径

### 6. 旧口径残留
- 旧规则数、旧路径、旧命名、旧 Action、旧字段名是否仍在系统文档中残留？
- 检查方法：`rg` 搜索已知的旧值（需从修改历史或当前不一致处推断旧值）

### 7. 验证闭环
- 是否有测试清单？
- 测试清单是否与当前实际行为一致？
- 是否包含让位测试 / 边界测试 / smoke test？

### 8. Decision Gate
- 输出强结论的 Skill 是否区分 signal / evidence / counter-evidence / decision？
- 是否按 claim type 定义 required evidence，而不是只列触发关键词？
- 是否定义 evidence completeness ceiling，限制 incomplete evidence 的结论等级？
- 证据不足时是否有 tentative / unresolved 降级路径？

### 9. 外围文档传播风险
- 即使主 SKILL 已修复，README / 手册 / 实战经验是否还在传播旧规则？
- 检查方法：在外围文档中搜索 SKILL.md 中已变更的关键词

---

## 设计初衷判断

如果系统最初有明确设计目标，额外判断：

```markdown
设计初衷：
- [原始目标 1]
- [原始目标 2]

判断：
- 结构上是否符合？
- 路由上是否符合？
- 实现上是否符合？

结论：完全符合 / 基本符合 / 部分偏离 / 明显偏离

若不符合 → 优先修什么才能重新对齐？
```

---

## 补充验证（多 Skill 系统专用）

### 1. 路由冲突扫描
- 比较所有 description / trigger phrase / DO NOT trigger 的交集
- 模拟常见用户输入，判断哪个 Skill 会被路由到

### 2. 残留口径扫描
- `rg` 搜索旧路径、旧规则数、旧字段名、旧 Action 名
- 覆盖范围：所有 .md 文件 + scripts/

### 3. 结构一致性扫描
- 检查多个 Skill 是否都具备同类必要模块
- 如：所有 Skill 都有触发条件？都有输出约束？

### 4. 测试清单一致性扫描
- 测试文档中的预期是否仍匹配当前实现

---

## 输出格式

按严重级别：
- **P0**：错误路由、错误结论、断链、错误执行的系统性问题
- **P1**：行为不稳、职责冲突、验证失真
- **P2**：维护成本上升、说明不准、文档漂移

报告包含：
1. 发现清单（含证据位置）
2. 已修复项（fix 模式）
3. 未修复风险
4. 验证方法
5. 是否满足原始设计初衷
