# 7 反模式脆弱性评估标准

> 用途：check/fix 模式加载，评估目标 Skill 对每种失效模式的防御能力。

## 评估方式

对每种反模式，检查目标 Skill 是否有**防御指标**。有防御 → 低风险，无防御 → 高风险。

---

## 模式 1：约束衰减（Instruction Decay）

**问题**：对话变长后 SKILL 规则逐渐失效。

**根因**：context window 压缩时，系统会摘要早期消息。SKILL 指令在 system prompt 中不被压缩，但用户消息中对规则的强化示例会被压缩，导致规则"权重"逐渐降低。中间大量工具输出会稀释 SKILL 指令在 attention 中的占比。

**防御指标**（有 = Low 风险）：
- ❗ 标记的核心规则（≤3 条）
- "每次输出前自检"机制
- "整个会话期间有效"显式声明

**脆弱性指标**（有 = High 风险）：
- 关键规则埋在长文本中无突出标记
- 无自检机制
- ❗ 标记超过 3 条（太多等于没标）

---

## 模式 2：工具选择漂移（Tool Selection Drift）

**问题**：该用专用工具时改用通用替代。

**根因**：两因素叠加 — (1) 熟悉度偏好：LLM 训练数据中 grep/cat 远多于特定 MCP 工具，倾向选择"更确定能用"的工具；(2) 单次失败过度泛化：一次 timeout 被误读为"这个工具不可用"。

**防御指标**：
- 工具优先级表（表格格式）
- 具体降级条件（"连续 N 次失败"而非"失败时"）
- 禁止未经确认的自行降级
- "单次超时 ≠ 工具不可用"规则

**脆弱性指标**：
- 无工具优先级表
- 模糊的"优先使用 X"表述
- 无降级协议

---

## 模式 3：输出膨胀（Output Bloat）

**问题**：生成大量冗余内容消耗 context。

**根因**：LLM 训练数据中"详细解释"比"简洁回答"获得更高人类评分。无明确长度约束时，默认策略是"多说总比少说好"。冗余输出消耗 context window，加速约束衰减（模式 1）。

**防御指标**：
- "禁止输出"列表（≥3 条具体项）
- 明确的长度/格式限制
- "不输出过渡句""不复述已知信息"等具体禁令

**脆弱性指标**：
- 仅有"保持简洁"等笼统要求
- 禁止项 <3 条
- 无领域相关输出限制

**链式影响**：输出膨胀 → context 填满 → 约束衰减(1) → 工具漂移(2)。控制输出量是阻断恶性循环的最有效位置。

---

## 模式 4：依赖链断裂（Dependency Chain Break）

**问题**：后续步骤未引用前序实际产出。

**根因**：LLM 每次生成本质上是"基于当前 context 的独立预测"，不会主动回溯之前的输出验证一致性。前序产出被 context 压缩后，后续步骤基于"模糊印象"而非精确数据生成。

**审计链防御**：用 Checkpoint 中的数字建立审计链 — 每步输出计数 → 下步引用上步计数 → 最终验证全链一致。

**防御指标**：
- 显式数据传递声明（"Step N 输入 = Step M 产出"）
- 禁止重新搜索/重新生成
- Checkpoint 计数链（后步引用前步计数）
- 写入前交叉验证

**脆弱性指标**：
- 步骤间无显式数据流
- 无计数传递
- 无写入前验证

---

## 模式 5：并行孤岛（Parallel Agent Isolation）

**问题**：并行子 agent 产出重复或矛盾。

**根因**：并行 agent 间没有通信机制，每个只看到自己的 prompt 和工具返回。主 agent 合并时倾向"把两份结果都放进去"而非"检查矛盾并选正确的"。

**并行化前置条件**：并行的前提是"任务可分且无依赖"。如果两个任务可能触及同一模块/数据，要么改为顺序执行，要么在 prompt 中明确划分边界（"你只分析包名以 X 开头的类"）。

**防御指标**：
- 明确分工边界（"你只负责 X"）
- 有数据依赖的任务标记为不可并行
- 合并检查清单（去重 + 一致性 + 计数验证）
- 矛盾处理规则

**脆弱性指标**：
- 职责重叠
- 无合并协议
- 简单拼接而非整合

**N/A 条件**：Skill 不使用子 agent → 标记 N/A。

---

## 模式 6：触发条件模糊（Trigger Ambiguity）

**问题**：Skill 误触发或漏触发。

**防御指标**：
- 多条件 AND 组合触发（非单一关键词）
- "不触发"反例（≥2 个场景）
- 歧义澄清问题
- 区分全链路触发 vs 单点触发

**脆弱性指标**：
- 单关键词触发
- 无"不触发"定义
- 无歧义处理
- 讨论/引用意图可能误触发执行

---

## 模式 7：幻觉填充（Hallucination Padding）

**问题**：工具返回空结果时编造内容。

**根因**：LLM 训练目标是"流畅且有帮助"，不是"忠实反映工具返回"。空结果时有两个选择：(a) 输出"未发现"— 简短、看起来"没做事"；(b) 基于领域知识推测 — 详细、看起来"深入分析"。训练偏好使 LLM 倾向 (b)。

**防御指标**：
- "所有结论必须引用来源"规则
- 零结果处理表（场景 → 正确输出 → 禁止输出）
- 结果分级标注（确认 / 降级 / 通用建议）
- "无来源 = 不输出"规则

**脆弱性指标**：
- 无来源引用要求
- 无零结果处理
- 允许"基于经验推测"作为结论

---

## 链式触发关系

```
输出膨胀(3) → context 填满 → 约束衰减(1) → 工具漂移(2)
                                                ↓
并行孤岛(5) → 矛盾数据 → 依赖链断裂(4) → 幻觉填充(7)
```

**典型恶性循环**：
1. 子 agent 输出膨胀（模式 3），返回大量冗余文本
2. 主 agent context 被占满，约束开始衰减（模式 1）
3. 主 agent 合并结果时未做一致性检查（模式 5）
4. 发现数据缺口时，用幻觉填充（模式 7）而非回溯修正
5. 最终产物看似完整，实际多处不一致或编造

**关键阻断点**：控制输出量（模式 3 防御）是阻断整条链的最有效位置。context 不被冗余内容填满 → 约束不衰减 → 后续模式触发概率大幅降低。

**评估规则**：
- 模式 3（输出膨胀）为 High → 模式 1 和 2 的实际风险自动上调一级
- 模式 5（并行孤岛）为 High → 模式 4 和 7 的实际风险上调一级

# 3 完整性原则检查标准

> 用途：check/fix 模式加载，评估目标 Skill 的完整性约束质量。
>
> 核心问题：LLM 读到"逐个处理"理解的是意图，不是约束。需要的不是描述意图，而是定义验收条件。

---

## 原则 1：可计数验收条件替代动词描述

**检查**：流程中可数产物的步骤是否用等式定义验收（`已完成数 == 应完成数`）？

**判定标准**：

| 级别 | 条件 |
|------|------|
| **Pass** | 所有可数步骤都有显式等式验收 |
| **Partial** | 部分步骤有等式，部分仍用"逐个处理""提取所有"等动词描述 |
| **Fail** | 无等式验收，全部使用动词描述 |

**检测方法**：
1. 找出流程中所有"产物数量可数"的步骤
2. 检查每步是否有 `X == Y` 形式的验收条件
3. 模糊写法示例（Fail）：`"逐个处理列表中的项目"` / `"按 N 条规则检查"` / `"提取所有 XXX"`
4. 可验证写法示例（Pass）：`"已处理数 == 列表总数"` / `"各结果计数之和 == N"`

---

## 原则 2：Checkpoint 切断"跳到结尾"路径

**检查**：每个步骤/阶段是否有强制中间输出（Checkpoint），阻止 LLM 一口气跳到最终输出？

**判定标准**：

| 级别 | 条件 |
|------|------|
| **Pass** | 每步有 ✅ Checkpoint + 量化指标（"Step N 完成: X 项"） |
| **Partial** | 部分步骤有 Checkpoint，部分直接跳到下一步 |
| **Fail** | 无 Checkpoint，步骤之间无强制输出 |

**检测方法**：
1. 查找 `✅ Checkpoint` 或类似格式的中间输出要求
2. 检查 Checkpoint 是否包含量化指标（而非仅"完成"二字）
3. 检查是否标注了"不可跳过"（↓ 【不可跳过】）

**关键理解**：Checkpoint 不是给人看的，是给 LLM 制造"不得不执行中间步骤的 token 生成义务"。没有中间输出 → LLM 没有锚点确认执行到哪一步 → 压缩或跳过。

---

## 原则 3：失败路径定义（无默默跳过空间）

**检查**：每个步骤是否定义了失败时的处理方式（显式 else），而非只有正常路径？

**判定标准**：

| 级别 | 条件 |
|------|------|
| **Pass** | 每步有失败降级路径：`操作A 失败 → 操作B → B也失败 → 记录原因，不允许跳过` |
| **Partial** | 部分步骤有失败处理，部分只有正常路径 |
| **Fail** | 无失败路径定义，LLM 的隐含 else 全是 skip |

**检测方法**：
1. 查找每步是否有"失败""降级""fallback"相关描述
2. 检查是否有"不可跳过"约束
3. 检查是否区分了"操作失败"和"步骤可跳过"

**关键理解**：任何没有显式 `else` 的步骤，LLM 的隐含 `else` 都是 `skip`。

---

## 评分汇总

| 原则 | Pass | Partial | Fail |
|------|------|---------|------|
| 1. 可计数验收 | 1.0 | 0.5 | 0 |
| 2. Checkpoint | 1.0 | 0.5 | 0 |
| 3. 失败路径 | 1.0 | 0.5 | 0 |

总分 /3 × 1.5 → 完整性维度得分（满分 1.5）
